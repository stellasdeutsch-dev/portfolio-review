The four lenses — full findings

Each reviewer rated every repo, filed severity-tagged findings with detail and fixes, and proposed concrete test cases.

image-search: strong video-analytics: strong image-classification: strong
Strong, idiomatic, well-factored code across all three repos; the core abstractions are right and the cores are genuinely tested — only the serving/glue layers have small, fixable smells.

These three repos are unusually disciplined for a portfolio: every project shares the same clean architecture — pure, dependency-light core logic (geometry, metrics, indexing, error analysis) separated from heavy I/O and ML stacks via lazy imports, explicit inter-stage data contracts in a dedicated schema.py, config-driven CLIs, and dependency injection at the seams (the Searcher's Encoder Protocol, the tracker/analytics split). That separation is the single best thing here: it makes the interesting logic testable without torch/faiss/cv2, and the tests actually exercise it. Naming is consistent, docstrings explain intent and trade-offs (not just mechanics), and error handling fails fast at stage boundaries with actionable messages. The weak spots are minor and concentrated in the serving/glue layers — a leaky lru_cache over mutable global state in the FastAPI app, an O(N*M) greedy tracker loop, and a few empty-input edge cases (np.mean([]) -> NaN) — none of which touch the well-tested cores. I verified the pure-core logic in all three repos runs correctly. This is the work of someone who understands why abstractions exist, not just how to use them.

Findings (7)
mediumlru_cache over mutable global STATE is a leaky, surprising abstraction— image-search
src/api.py:71-74 defines `@lru_cache(maxsize=512) def _cached_text_search(version, used, k)` whose body reads the module-global `STATE["searcher"]`. The cache is keyed on a manually-bumped `version` int (incremented in the lifespan handler), but the function closes over mutable global state rather than being a pure function of its arguments. Consequences: (1) entries for stale `version` values are never purged on reload — they sit in the cache until LRU-evicted, holding references to old result lists; (2) the cache is process-global and survives across the lifespan teardown that does `STATE.clear()`, so a function that now reads a cleared STATE could KeyError if ever called post-shutdown; (3) it's hard to reason about and untested. This is the one place in the codebase where the abstraction fights the code.
→ Replace the module-level lru_cache with an explicit per-searcher cache (e.g. a `functools.lru_cache`-wrapped bound method created in Searcher.__init__, or a small dict on the Searcher keyed by (used, k)). That ties cache lifetime to the object it caches for, makes invalidation a matter of dropping the Searcher, and removes the reliance on a hand-incremented version counter.
lowGreedy tracker association is O(N*M) per match via repeated np.argmax— video-analytics
SimpleTracker.update (src/track.py:49-56) does greedy IoU matching by `np.argmax` over the full (T,D) IoU matrix, then masks the chosen row/col to -1 and re-scans the whole matrix each iteration — O(T*D) per match, O(min(T,D)*T*D) overall. For the smoke-test frame sizes this is irrelevant, but the module docstring positions it as swappable with ByteTrack/BoT-SORT, and greedy IoU is known to be suboptimal vs. Hungarian assignment (it can make locally-greedy matches that strand a better global pairing). It's a correctness-of-results smell more than a performance one.
→ Either (a) note explicitly in the docstring that association is intentionally greedy and the production path is the Ultralytics tracker, or (b) swap the inner loop for scipy.optimize.linear_sum_assignment on the IoU cost (guarded by an import check to preserve the dependency-free promise). The current greedy version is fine for the demo; the fix is about matching the stated ambition.
lowEmpty-input paths produce np.mean([]) -> NaN with a RuntimeWarning— image-search
eval/evaluate.py:48-69: if the queries file is empty, `recalls[k]`, `aps`, and `latencies` are all empty lists, and `np.mean([])` / `np.percentile([], 50)` emit a RuntimeWarning and return NaN, which then serialize into report.json as `NaN` (invalid JSON). The same NaN-on-empty pattern appears in video-analytics analytics.py:123 (`np.mean(dwell_vals)` is guarded, good) but the avg_speed at line 143 is guarded too — eval is the unguarded one.
→ Guard the aggregation: `if not items: raise SystemExit('no queries') ` or return a report with explicit 0.0/null values and `n_queries: 0`. At minimum short-circuit before the np.mean/np.percentile calls so the output stays valid JSON.
lowannotate() implicitly assumes track frame indices align with cap.read() order from 0— video-analytics
src/annotate.py:26-46 builds `by_frame = {f: g for f,g in tracks_df.groupby('frame')}` then iterates `cap.read()` with a local `idx` starting at 0 and looks up `by_frame.get(idx)`. This silently assumes the detector enumerated frames 0..N-1 contiguously in the same order OpenCV will re-read them. That holds for the current detect.py (which uses `enumerate(results)`), but it's an undocumented coupling between two modules that don't share a contract — if a future detector skips frames or starts at 1, boxes render on the wrong frames with no error.
→ Make the coupling explicit: document the frame-index contract in schema.py alongside DETECTION_COLUMNS/TRACK_COLUMNS, or have annotate read the frame index from the video (cap.get(CAP_PROP_POS_FRAMES)) rather than a local counter, so it's robust to non-contiguous frame numbering.
lowDuplicated inverse-frequency class-weight logic (train.py vs data.py)— image-classification
The inverse-frequency weight formula `counts.sum()/(len(counts)*counts)` appears twice: src/data.py:91-100 (class_weights, computed over the whole dataset folder) and src/train.py:51-56 (computed over the train split only, correctly avoiding leak). train.py deliberately recomputes from the train indices rather than calling data.class_weights — which is the right call for correctness — but that leaves data.class_weights as a near-identical, leak-prone function that nothing uses and which a future caller could reach for by mistake.
→ Either delete the unused data.class_weights, or refactor both to call one helper that takes explicit per-class counts (e.g. `inverse_freq_weights(counts)`), so the leak-safe behavior lives in one place and the dataset-wide variant can't be wired in accidentally.
infoTracker `tsu` (time-since-update) lifecycle has an off-by-one worth a comment— video-analytics
In SimpleTracker.update (src/track.py:68-80), unmatched tracks get `tsu += 1`, then the prune keeps `tsu <= max_age`. A track matched this frame has tsu==0 and is emitted only if `hits >= min_hits` (line 84). The semantics are correct and the test suite covers expiry/reappearance, but the interaction of max_age (frames kept alive while lost) and min_hits (frames before first emit) is subtle and only partially documented. The docstring explains the sparse-detector gap but not the lifecycle constants.
→ Add one or two lines to the SimpleTracker class docstring stating precisely what max_age and min_hits mean in frame units and how they interact, since these are the knobs a reader will most want to tune.
infoscripts/* and web/* sys.path manipulation is a minor packaging smell— image-search
web/app.py:12 and web/explorer.py:15 (image-classification) do `sys.path.insert(0, ...)` to import the src package when run directly. This works but is the kind of thing that bites later (import order, running from a different cwd). Since each repo already ships a pyproject.toml with the package declared, `pip install -e .` makes these unnecessary.
→ Document in the README that the web UIs require `pip install -e .` first, and drop the sys.path hack — or keep it but guard it behind a check that the package isn't already importable, so an installed environment uses the real package.

image-search: strong video-analytics: solid image-classification: solid
Solid portfolio-grade engineering with genuinely good test seams, but column-only schema validation plus dev-only CI means malformed inputs and the entire real ML/serve path are unguarded — fix value-range validation, add weights_only to torch.load, validate API k, and add a heavy-deps smoke job before calling any of these release-ready.

These are well-structured portfolio repos with a genuinely good QA instinct: explicit inter-stage data contracts (src/schema.py in each), pure-NumPy cores that are tested without the heavy ML stack, and dependency-injection seams that make retrieval/analytics/metrics unit-testable. The happy paths and several edge cases (k>corpus, empty detections, empty trajectories, perfect classifier) are handled cleanly. The release-readiness gap is consistent across all three: the schema validators check column presence only, never dtypes or value ranges, so malformed-but-well-shaped artifacts (out-of-range labels, NaN/out-of-range confidences, negative k) sail past validation and either crash with an opaque error three stages later or silently produce wrong numbers. Untrusted-input handling is uneven: the image-search upload path is hardened (size cap, decode guard), but torch.load on a checkpoint and the LLM/web layers have soft spots. None of this is alarming for a portfolio, but several items would block a real release.

Findings (12)
highSchema validators check columns only, not dtypes or value ranges — malformed-but-shaped artifacts crash downstream— all repos
All three src/schema.py validate column presence and nothing else. A predictions.parquet with the right columns but a stale/out-of-range label, a manifest with non-integer width, or an embeddings array with wrong dtype passes validation and fails three stages later. Confirmed in image-classification: confusion_matrix([0,1,2],[0,1,7],3) raises 'IndexError: index 7 is out of bounds' (src/metrics.py:15 np.add.at), and a negative label silently wraps (cm.sum stays 6, wrong cell incremented). The contract promises 'fail fast at the boundary with a clear message' but only delivers that for missing columns.
→ Extend each validator to assert dtype and value ranges at the boundary: predictions y_true/y_pred in [0, n_classes), confidence in [0,1] and finite; manifest width/height integer and >0; detections/tracks coordinate columns finite numeric. Raise the same ValueError style as the column check so producers fail at write time, not consumers at read time.
highCI never exercises the real ML/serve path — green CI only proves the NumPy fallback— all repos
.github/workflows/ci.yml in each repo runs `pip install -e .[dev]` then pytest; the dev extra is pytest+ruff (+fastapi/httpx for repo 1) but NOT torch/faiss/timm/cv2/ultralytics. So Searcher.from_config (ClipEncoder+FAISS), predict.run (torch), detect.run (ultralytics), annotate (cv2), and Grad-CAM are never run in CI. test_gradcam uses pytest.importorskip('torch') and is silently skipped. A regression in any real-model wiring (e.g. a wrong open_clip arg, a checkpoint key rename) ships green.
→ Add a second CI job (allowed-to-fail / nightly) that installs the heavy extra on CPU and runs at least one true end-to-end smoke per repo on the sample data the make_sample scripts already generate. At minimum, fail CI if the torch-dependent tests are all skipped (collected==skipped).
hightorch.load on checkpoint without weights_only=True is arbitrary code execution— image-classification
src/predict.py:28 `ckpt = torch.load(cfg['checkpoint'], map_location=device)` deserializes a pickle with no weights_only guard. A checkpoint path is config-driven and could come from a shared cluster scratch dir, a teammate, or a downloaded artifact; loading a tampered .pt executes arbitrary code. Classic untrusted-deserialization vector.
→ Pass weights_only=True (torch>=2.x) when loading, and load class_names/arch from the separate classes.json rather than trusting pickled metadata. If full-object load is unavoidable for older checkpoints, gate it behind an explicit config flag and document the trusted-source requirement.
mediumAPI search k is unvalidated — negative/zero k silently returns wrong results— image-search
TextQuery.k is `int | None` with no bounds; search_text uses `k = body.k or s.top_k` and passes it to index.search. The NumPy path does `k = min(k, N)` then slices `[:k]`, so I confirmed k=-1 on a 3-vector index returns 2 results (silently wrong) and k=0 returns an empty list with status 200 (looks like 'no matches'). A huge k just clamps. None are rejected; the LLM-cached path also keys lru_cache on the raw k.
→ Validate k with pydantic (k: int | None = Field(default=None, ge=1, le=MAX_K)) and reject out-of-range with 422. Mirror the clamp/validate in VectorIndex.search so the library itself refuses k<1.
mediumLine-crossing counter over-counts trajectories that graze the counting line— video-analytics
count_crossings + segments_intersect (src/common.py:111) use strict orientation tests. A trajectory that touches the line and returns to the same side can still register a crossing: I ran traj [5,2]->[5,0]->[5,2] against line y=0 and got (forward=1, backward=0) even though the object never reached the far side. The segment starting exactly on the line (orient==0 at p1) makes the (d1>0)!=(d2>0) test fire. For traffic counting this inflates counts at objects hovering near the line (slow/queued vehicles).
→ Encode a consistent on-line policy: count a crossing only when a point is strictly on the opposite side after being strictly on one side (track last strict side per trajectory), or use a half-open orientation rule. Add a regression test for grazing/touch-and-return trajectories.
mediumcalibration_bins silently drops out-of-range/NaN confidences but divides ECE by full n— image-classification
Bins only cover (0,1]; confidence values >1, <=0, or NaN fall into no bin (confirmed: input [1.5,-0.2,0.5] yields one bin of count 1; [nan,0.5] yields count 1). But ECE accumulates `(cnt/max(n,1))*|acc-conf|` where n=len(confidence) includes the dropped samples, so the reported ECE is scaled by a denominator larger than the points actually binned — a quietly wrong calibration number with no warning. Softmax won't produce these, but predictions.parquet is an explicit, hand-editable contract other producers could feed.
→ Validate confidence in [0,1] and finite at the schema boundary (see finding 1), or compute the ECE denominator from the number of binned samples and log a warning when any confidences are dropped.
mediumSimpleTracker.update hard-crashes on any non-6-column detection row— video-analytics
src/track.py:42 `dets = np.asarray(dets, dtype=float).reshape(-1, 6)`. A single malformed row (5 or 7 values) raises 'cannot reshape array of size N into shape (6)' (confirmed) and aborts the entire track stage for the whole video.
